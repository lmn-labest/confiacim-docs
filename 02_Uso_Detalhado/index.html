Na pasta simulation deve ficar os arquivos de entrada. Na caso já temos três exemplos simples na pasta. No case1a a médias da variaveis são consideras unitárias. Os valores reais das médias estão nos arquivos de template. Já no case1b as médias estão definidas direramente no arquivo yml. Mas os dois são a mesma simulação com arquivos de entradas diferentes. Já case8a é uma simulação completamente diferente.

Os arquivos de entradas são os mesmo do arquivos do tencim1D com mais alguns arquivos extras *.jinja e case.yml. Na pasta templates estão os arquivos de entrada que serão gerados dinamicamente, esses arquivos precisam ter extensão *.jinja. No arquivo case.yml estão as configurações da simulação do confiacim. No campo variables são definidos as variáves que serão utilizadas, é necessario que existam pelo menos uma variavel na lista. No campo correlations são definidas as correlações entre as variáveis, este campo é opcional, caso não seja fornecido esse campo a matriz de correlação é automaticamente definida como a matriz identidade. No campo config temos as configurações do confiacim.

2) Análise probablística com FORM

Para rodar basta fazer:

1
confiacim form run simulation/case1a/

Para uma saida mais verbosa basta utilizar a opção --verbose ou -v:

1
confiacim form run --verbose simulation/case1a/

Tip

Opcionalmente pode-se definir explicitamente o diretório de saida como:

1
confiacim form run simulation/case1a/ simulation/case1a/output2

Após rodar a estrutura de diretórios deve ser algo similar:

Na pasta output/form estão salvos os resultados desejados da execução do FORM. Os resultados do FORM estão no arquivo results.json.

Para salvar os arquivos RCs e outros arquivos intermédiarios basta usar a opção -sou --save:

1
confiacim form run -s simulation/case1a/

Por exemplo, na pasta it_0001 estão os resultados da iteração 1 do FORM. Com os arquivos intermédiarios são salvos. O confiacim e capaz plotar algums gráficos. Para plotar todos os resultados de RC do FORM:

1
confiacim form plot all simulation/case1a/output/

Fig1 - Série histórica do valor do RC agrupados por iterações.

Para plotar os resultados de RC de uma iteração específica:

1
confiacim form plot it 1 simulation/case1a/output/

Fig2 - Serie histórica do valor do RC da iteração 1.

Para salvar os gráficos pode-se usar a opção -s ou --save

1
confiacim form plot all -s simulation/case1a/output/

Os gráficos serão salvos na pasta simulation/case1a/output/run1/graphs.

Para um gráfico mais iterativo existe a possibilidade de usar o plotly:

1
confiacim form plotly all simulation/case1a/output/

3) Análise probablística com Monte Carlo

Para rodar basta fazer:

1
confiacim mc run simulation/case1a/

Para uma saida mais verbosa basta utilizar a opção --verbose ou -v:

1
confiacim mc run --verbose simulation/case1a/

Tip

Opcionalmente pode-se definir explicitamente o diretório de saida como:

1
confiacim mc run simulation/case1a/ simulation/case1a/output2

Para salvar os arquivos RCs e outros arquivos intermédiarios basta usar a opção -s ou --save:

1
confiacim mc run -s simulation/case1a/

No Monte Carlo as opções de configuração são:

1
2
3
4
5
config:
  monte_carlo:
    n_samples: 10000
    batch_size:  100
    seed: 1234

Sendo elas:

  • n_samples: Número de amostras;
  • batch_size: Número de lotes das amostra que serão rodada. Por exemplo se esse valor for 100 o monte carlo ira rodar de 100 e 100 amostras até completar o número total de amostras;
  • seed: Semente usada para gerar as semenete usadas nas distribuições uniformes.

No Monte Carlo caso o Tencim1D falhe a simulação continua. Os arquivos de entrada e saida do caso que falhou serão salvos na pasta failure. O cálculo da probabilidade é calculada usando apenas os casos que realmente rodaram.

A saida são os arquivos results.json o batch_results.json. A informações do arquivo results.json são:

O campo total é o número total de amostras, valid_simulations são as simulações do Tencim1D que rodaram até o fim, fail é o número de amostra que com falha estrutural, P é a probabilidade de falha e estimator é o estimador de convergencia do monte carlo. 4) Análise deterministica

Para rodar a análise determinística base:

1
confiacim deterministic run simulation/case1a

Os resultado ficam na pasta output/deterministic.

Tip

Opcionalmente pode-se definir explicitamente o diretório de saida como:

1
confiacim deterministic run simulation/case1a/ simulation/case1a/output2

Para plotar os RCs da análise deterministica:

1
confiacim deterministic plot  simulation/case1a/output/

Para truncar a análise em 184 basta:

1
confiacim deterministic new_last_step 184 simulation/case1a/case.dat

Será gerado um novo case.dat. Será criado um case_original.dat com o conteudo original do arquivo case.dat.

Caso se queira agora truncar a análise em 139 você poded usar o case_original.dat, ou seja:

1
confiacim deterministic new_last_step 139 simulation/case1a/case_original.dat

Para voltar para o original basta usar step maior que o original, por exemplo:

1
confiacim deterministic new_last_step 10000 simulation/case1a/case_original.dat

5) Configurações

Algumas configurações extras podem ser feitas através do arquivo .env. Este arquivo deve estar na raiz do projeto, porem ele é completamente opcional.

1
2
3
4
5
6
7
8
9
CONFIACIM_EXEC_TENCIM_BIN='TENCIM1D.exe'
CONFIACIM_EXEC_TENCIM_DIR='/opt/confiacim/'
CONFIACIM_EXEC_TENCIM_MODE='SEQUENTIAL'
CONFIACIM_LOG_LEVEL='DEBUG'
CONFIACIM_HIDRPROP_FILE_NAME='hidrationprop.dat'
CONFIACIM_MATERIALS_FILE_NAME='materials.dat'
CONFIACIM_INITIAL_TEMP_FILE_NAME='initialtemperature.dat'
CONFIACIM_LOADS_FILE_NAME='loads.dat'
CONFIACIM_LOG_FILE_PATH='/var/log/confiacim.log'

A variável de ambiente CONFIACIM_LOG_LEVEL irá definir o nível de informações que serão escritos no aquivo conficacim.log e simulation.log. Atualmente existem 3 niveis ERROR, INFO e DEBUG.

A váriavel CONFIACIM_EXEC_TENCIM_MODE define o modelo de excução do Tencim1D. As opções são: SEQUENTIAL parar rodar o tencim1D em sequencial; MULTIPROCpara rodar utilizando multiprocessos. O valor padrão é o MULTIPROC. O número de processos será o menor valor entre número de variaveis + 1 e o número de n. Onde CPUS retornado pelo função os.cpu_count() dividido por 2.

Outra forma de alterar os valores configurações é passando elas na hora de executar o confiacim. Por exemplo para mudar o nível do log pode-se fazer:

1
LOG_LEVEL="ERROR" form form run simulation/case1a

Nessa caso a variável passada na linha de comando irá prevalecer sobre a variável que está no arquivo .env.

Geralmente o arquivo .env é usado em ambiente de desenvolvimento. Em produção o melhor é definir a variável de ambiente. Porém nada impede dele ser usado em ambiente de produção caso se queira.

6) Executável do tencim

Por padrão existem dois executáveis pré-compilados do tencim na pasta data/prebuild na instalação padrão do confiacim. Automaticamente o confiacim identifica se o sistema é windows ou linux e usa o binário certo. Porém caso seja necessário fornecer outro tencim pode-se passar caminho completo atravéz da variável de sistema CONFIACIM_EXEC_TENCIM_DIR. Também é possivel trocar o nome do executável usando CONFIACIM_EXEC_TENCIM_BIN. Por exemplo caso o tencim esteja instalado na pasta /opt com o nome de tmc1d.exe. Basta configurar as seguintes variáveis de sistema.

1
2
CONFIACIM_EXEC_TENCIM_BIN="tmc1d.exe"
CONFIACIM_EXEC_TENCIM_DIR="/opt/"

7) Arquivos de saida

Existe duas opções de arquivos de saidas. O formato .json e .txt. O padrão é o json. Para mudar o formato do arquivo existe a chave de configuração result_files. Por exemplo para usar a saida txt basta:

1
2
3
4
5
6
7
8
config:
  result_files: "txt"
  form:
    beta: 20.0
    delta: 0.01
    gamma: 0.3
    tol:  1e-01
    maxit: 2

Para a saida json ainda existe a chave de configuração json_indent. Essa configuração define o espaçamento no arquivo json. Por padrão a indentação é zero, isso deixa o arquivo menor porém menos legível. Para aumentar, para por exemplo 4 espaços, basta:

1
2
3
4
5
6
7
8
config:
  json_indent: 4
  form:
    beta: 20.0
    delta: 0.01
    gamma: 0.3
    tol:  1e-01
    maxit: 2

9) Selecionando critéros de falha

Atualmente existem dois criterios de falha. Por Rankine e Mohr-Coulomb. O padrão é o Mohr-Coulomb. Para a configurar o critério basta usar a chave rc_criteria. Essa chave é opcional.Para usar o Rankine basta:

1
2
config:
  rc_criteria: rankine

Para usar o Mohr-Coulomb não é preciso informar nada pois ele já é o padrão, porém pode-se defini-lo explicitamente também como:

1
2
config:
  rc_criteria: mohr-coulomb
